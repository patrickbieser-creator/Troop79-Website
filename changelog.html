Running the Production App (Next.js + Supabase)

Files live in D:\Projects\Troop79-Website\next-app\ — the real, database-backed application you are actively building. This is not the static prototype below.

The app talks to a local Supabase database that runs inside Docker containers. Data (scouts, ledger, merit badges) persists in Docker volumes between sessions — starting up does not reseed or wipe it.

Everyday startup — resuming work

  • 1. Start Docker Desktop and wait until it reports running (whale icon steady in the tray). The Supabase database can’t come up until Docker’s engine is ready.
  • 2. In a terminal: cd D:\Projects\Troop79-Website\next-app
  • 3. npx supabase start — boots the Postgres + API containers (~15–30 s when images are already cached). Your existing data is still there.
  • 4. npm run dev — starts the Next.js dev server.

Then open in a browser:

  • http://localhost:3000/ — the live app (public site)
  • http://localhost:3000/admin/advancement/dashboard — Leader Workspace / admin (requires leader login)
  • http://127.0.0.1:54323 — Supabase Studio: browse & query the active database directly (verify rows, run SQL)

Shutting down

Ctrl+C stops the dev server. npx supabase stop stops the database containers — data is preserved on the volume. Quitting Docker Desktop stops everything.

First-time setup or a clean database

Only needed on a fresh machine or when you want to start the DB over. Keys: npx supabase status -o env and paste the output into next-app\.env.local. Schema + data: npx supabase db reset (applies migrations, wipes to empty), then npm run seed (reference taxonomy: leaders, ranks, merit-badge catalog — not scouts/ledger), then npm run import-spreadsheet -- --xlsx="…" --apply to load scouts + ledger. A plain everyday startup skips all of this.

Running the Static Prototype (design reference)

Stop the server with Ctrl+C. Port can be any free number — 8000 is just convention. On this machine use py (the Python launcher); a bare python is currently shadowed by a Microsoft Store stub. npx serve . works too (Node 24+ installed).

v0.19 July 6, 2026 New

News & Events CMS — Architecture Decision

  • The original build spec for a News & Events publishing system assumed a brand-new Payload CMS project with its own Postgres database and deploy pipeline. It wasn’t green-field: index.html at the repo root already is the News/Events design reference, and next-app/ already has a working Supabase database, a leader-session auth pattern, and an admin workspace to extend. Decided against Payload — the feature is being built directly into the existing Next.js + Supabase stack instead of standing up a second backend to maintain. The one real tradeoff: Payload’s admin panel generates a polished drag-and-drop rich-text block editor for free. Building that by hand would have been the single biggest chunk of effort in the project — solved instead by authoring in markdown (already a comfortable format) plus a small set of custom content blocks for the few things markdown has no native syntax for.
  • Media hosting: Bunny CDN, the troop’s existing account, via a server-side upload proxy so the API key never reaches the browser. Google Photos and Facebook albums are not embedded — Google Photos has no supported embed API and scraping its app is fragile, so a new Gallery Link content block renders a styled link-out card (cover photo, source badge, “View Full Album” button) instead of pretending to embed something that can’t be embedded.
  • Interactive wireframes for all three article templates (News, Event, Recognition), the split-pane markdown editor, the media picker, and the Gallery Link card were built and browser-verified first, at prototypes/news-cms/ — the same “prototype before production” process used everywhere else in this project.

Data Model & Auth

  • New tables: articles (one row per News/Event/Recognition post, discriminated by type — same single-table shape as ledger_entries), media (Bunny-hosted images with required alt text), tags, and the article_tags join table. Archiving an article soft-hides it from the public site without deleting it, matching the ledger’s existing archive convention.
  • The stub leader-session cookie now carries a role (leader or scout) so “scouts draft, leaders publish” is actually enforced: a scout can create and edit their own drafts but every publish attempt is rejected server-side; a leader can publish, archive, or delete anything. Existing signed-in sessions default safely to leader so nobody was logged out by the change.

Markdown Body & Content Blocks

  • Article bodies are authored in markdown. Standard syntax covers prose, links, lists, images (![alt](url "caption")), and tables — no custom syntax needed for any of those. Three things markdown has no native construct for get a small custom {{type: ...}} token, inserted automatically by editor toolbar buttons rather than hand-typed: Gallery (an inline photo grid), Gallery Link (the external-album card), and Video (a YouTube/Vimeo embed with a graceful fallback card for unrecognized links).
  • Three real rendering bugs were caught and fixed during browser verification, not just eyeballed: markdown’s auto-linking of bare URLs was silently splitting every real gallery/video token so it never matched (fixed by matching against the original source text instead of the parsed fragments); the token parser and the React renderer disagreed on property names, so a matched token would have rendered as an empty box; and a markdown image renders as block-level HTML but is always wrapped in a paragraph tag by the parser, which is invalid nesting and triggered a real hydration error in the browser.

Admin — Article Authoring

  • New Articles table under a “News & Events” workspace section — filter by type/status/tag, per-row Publish (leader-only), Archive, and Delete, plus a leader-only Featured toggle for pinning a small set of posts to the top of the future home page.
  • New two-pane article editor, split-screen like the prototype: form fields (title, type, hero image, excerpt, tags, event date/time/location for Event-type posts) on the left, a live preview of the actual rendered article on the right, updating as you type. Insert Image / Insert Gallery / Insert Gallery Link / Insert Video toolbar buttons open the media picker and write the right markdown or content-block token at the cursor automatically.
  • New leader-only Tags page (add / delete) rounds out the admin side. Ten starter tags are seeded (Advancement, Camping, Service Project, Fundraising, Court of Honor, Merit Badges, Community, High Adventure, Meetings, Recruiting).

Still to come

  • Open Public-facing pages — home/news feed, article detail, events list, tag archive — not built yet. The admin side above is fully usable but nothing is visible to the public until these ship.
  • Open Real Bunny CDN credentials haven’t been wired in yet, so live photo upload/browse hasn’t been tested end-to-end against the troop’s actual account — the code path was verified with placeholders.
v0.18 July 6, 2026 Feature

Admin — Deterministic Audits Section

  • New Audits workspace section (replacing the earlier single-purpose Rank Audit page), built to host multiple independent, non-AI checks side by side. Two ship today: Board of Review done but requirements missing (a scout has a Board of Review on record for a rank but is missing some of that rank’s own requirement rows) and Activity / Campout thresholds (a scout has logged enough activities to satisfy Second Class 1a — 5 activities / 3 campouts — or First Class 1a — 10 / 6 — but doesn’t have that requirement signed off yet). The two checks are deliberately complementary: BoR-missing is reactive (only catches a gap once the rank is already “complete”), Activity/Campout is proactive (fires from the ledger alone, regardless of BoR status). A scout can reasonably appear in both for the same gap — two independent confirmations, not a bug.
  • Every finding gets a review card — a checkbox list of the missing requirements plus date + signing-leader inputs. Approving inserts the missing rows straight into the Universal Ledger; the card disappears on refresh.
  • Bug fix: both checks now correctly exclude inactive scouts (a scout who dropped out or moved away no longer shows up as “missing” a requirement they’ll never complete).

Universal Ledger & Fast Entry — Event Classification

  • Fixed a real miscategorization bug: the annual Pancake Breakfast fundraiser — 55 rows across three years — had been logged as a 2-night campout, simply because Fast Entry had no way to say “this event isn’t a campout or a hike.” Every event now carries its own Type — Campout, Hike, Day Outing, or Fundraiser — corrected in place for the 55 affected rows after a dry-run review.

A recurring Fundraiser or Day Outing logs itself with zero extra clicks; a Campout or Hike still asks for Nights/Miles as usual, with Type only stepping in if a particular visit doesn’t fit the event’s usual pattern. All 28 existing events were classified automatically from their own history — nothing was left unclassified. Design went through three rounds in one sitting: first a separate editable “Event Type” lookup table alongside the existing ledger type; then Camping/Hiking were recognized as redundant with the existing type field and dropped from it; then the whole separate lookup table was recognized as redundant too and folded directly in — every event category is now a single, first-class ledger type, with no translation layer in between. The generic “Attendance” catch-all is gone entirely — every check-in event resolves to a real category.
  • The Universal Ledger’s Type column and filter, the Leader Dashboard’s recent-activity feed, and the Scout Clipboard’s Activities panel all display the corrected categories consistently.

Step-by-Step Procedure

  • 1. Start Docker Desktop and wait for the whale icon to stop animating. Supabase CLI uses Docker for the local Postgres + PostgREST + Auth + Studio stack — nothing happens until Docker is ready.
  • 2. From next-app/, run npx supabase start. First boot pulls ~1.5 GB of container images (postgres, gotrue, postgrest, studio, kong, edge-runtime, …) — takes 5–15 minutes on first run, <30 seconds thereafter. Subsequent boots are fast because images are cached.
  • 3. Capture the keys with npx supabase status -o env. Easier than copy/pasting from the multi-panel ASCII art the start command prints. The -o env flag emits them as KEY="value" pairs. Note: newer Supabase CLI also prints PUBLISHABLE_KEY + SECRET_KEY (the new naming). For now we use the legacy ANON_KEY + SERVICE_ROLE_KEY — both formats work, @supabase/ssr accepts either.
  • 4. Create next-app/.env.local with three lines: NEXT_PUBLIC_SUPABASE_URL (the API URL, typically http://127.0.0.1:54321), NEXT_PUBLIC_SUPABASE_ANON_KEY (the long JWT-shaped anon key), and SUPABASE_SERVICE_ROLE_KEY (the long service-role JWT). The file is already in .gitignore. Template lives at .env.example.
  • 5. Apply the schema with npx supabase db reset. The first supabase start on an empty volume sometimes does not auto-apply migrations — symptom is Could not find the table 'public.ledger_entries' on the next step. db reset wipes & recreates with the migration applied. Safe to repeat any time you want a clean DB.
  • 6. Seed from prototype JSON: npm run seed. Reads ../data/advancement.json and inserts 18 leaders, 9 activity types, 7 ranks, 171 rank requirements, 32 merit badges, 89 MB requirements, 9 scouts, 52 ledger entries, 2 COH history rows. Idempotent — truncates first.
  • 7. Start the dev server: npm run dev. Open http://localhost:3000/merit-badges. Supabase Studio (DB browser) lives at http://127.0.0.1:54323 — use it to verify rows after seeding.

Fix: Seed Script Now Reads .env.local

  • npm run seed changed from tsx scripts/seed.ts to tsx --env-file=.env.local scripts/seed.ts. tsx does not auto-load .env* files the way next dev does, so the seed script was reading process.env.SUPABASE_SERVICE_ROLE_KEY as undefined even with the key correctly written to .env.local. Node’s built-in --env-file flag (Node 20+) is the cleanest fix — no dotenv dependency needed.

Recovery Recipes

  • “Container name already in use” error when running supabase start → a previous container is stopped but not removed. Fix: npx supabase stop --no-backup, then npx supabase start again. The --no-backup flag avoids prompting about the volume.
  • “Could not find the table” on seed/dev → migration didn’t apply. Fix: npx supabase db reset. Re-run npm run seed afterwards.
  • Need to wipe and start fresh: npx supabase db reset for schema-only reset (preserves containers); npx supabase stop then start for full teardown.
  • Keys rotated or forgotten: npx supabase status -o env any time — the local keys are deterministic per project and do not change between boots unless you delete the volume.

Universal Ledger — Built for Scale

  • Pagination: 100 entries per page by default, with a selector for 50 / 100 / 250 / 500 / 1000. Pager controls (First, Prev, Next, Last) plus a jump-to-page input and a “Showing X–Y of Z” counter. Measured on a 5,010-entry prototype: filter keystroke responds in ~5 ms; jump-to-last-page in ~1 ms. At 50K entries the table is still snappy in a single browser tab; past that we’d move filter/sort/paging to the server.
  • Sortable columns: click any column header (Date, Scout, Type, Code, Description, Signed Off, Qty, Unit, Entered) to toggle ascending/descending. Active column shows an arrow indicator. Date/Entered/Qty default to descending; text columns default to ascending. Qty sorts numerically; Date sorts on the ISO string so months order correctly.
  • Archive (lifecycle) and Delete (erroneous) are now distinct row actions:
    • Archive — one-click, restorable. For entries that are still valid but no longer belong in the active view (e.g. scout aged out, periodic cleanup).
    • Delete — soft-delete that requires a typed reason. For erroneous entries (duplicate, wrong scout, typo). The reason is stored on the entry and shown on hover.
    Both retain audit metadata (archivedAt/archivedBy or deletedAt/deletedBy/deletedReason) and are recoverable via the new “Show hidden rows” toolbar toggle. Archived rows render greyed-out and striped; deleted rows additionally show strikethrough with a red marker.
  • Bulk archive by scout (aged-out workflow): the Lookups & Admin screen now has a per-scout Mark aged-out button. Confirms the action, then archives all of that scout’s ledger entries in one pass with archivedReason: scout_aged_out and flips scout.active = false. A Reactivate & restore button reverses it, restoring only the entries archived for that reason — entries archived manually stay archived.
  • Dashboard counts reflect active entries only: the sidebar badge and the “Ledger Entries” stat now exclude archived and deleted rows. Recent Activity on the dashboard skips them too.
