Running the Prototype Locally

The site is plain static HTML — no build step. Serve the project root with Python’s built-in HTTP server:

cd D:\Projects\Troop79-Website
python -m http.server 8000

Then open any of these in a browser:

  • http://localhost:8000/ — Home / News Feed
  • http://localhost:8000/calendar.html — Events & Calendar
  • http://localhost:8000/meeting.html — This Week’s Meeting
  • http://localhost:8000/advancement.html — Advancement (troop roster)
  • http://localhost:8000/advancement-scout.html?id=maya — Scout Clipboard (drill-down)
  • http://localhost:8000/admin/meeting-editor.html — Meeting Editor (admin)
  • http://localhost:8000/admin/advancement-leader.html — Advancement Leader Workspace (admin)
  • http://localhost:8000/changelog.html — This changelog

Stop the server with Ctrl+C. Port can be any free number — 8000 is just convention. If python isn’t on PATH, npx serve . works too (Node 24+ installed).

v0.7 May 25, 2026 Feature

Universal Ledger — Built for Scale

  • Pagination: 100 entries per page by default, with a selector for 50 / 100 / 250 / 500 / 1000. Pager controls (First, Prev, Next, Last) plus a jump-to-page input and a “Showing X–Y of Z” counter. Only the visible page is rendered to the DOM, so the table stays responsive at tens of thousands of entries. Measured on a 5,010-entry prototype: filter keystroke responds in ~5 ms; jump-to-last-page in ~1 ms. At 50K entries the table is still snappy in a single browser tab; past that we’d move filter/sort/paging to the server.
  • Sortable columns: click any column header (Date, Scout, Type, Code, Description, Signed Off, Qty, Unit, Entered) to toggle ascending/descending. Active column shows an arrow indicator. Date/Entered/Qty default to descending; text columns default to ascending. Qty sorts numerically; Date sorts on the ISO string so months order correctly.
  • Archive (lifecycle) and Delete (erroneous) are now distinct row actions:
    • Archive — one-click, restorable. For entries that are still valid but no longer belong in the active view (e.g. scout aged out, periodic cleanup).
    • Delete — soft-delete that requires a typed reason. For erroneous entries (duplicate, wrong scout, typo). The reason is stored on the entry and shown on hover.
    Both retain audit metadata (archivedAt/archivedBy or deletedAt/deletedBy/deletedReason) and are recoverable via the new “Show hidden rows” toolbar toggle. Archived rows render greyed-out and striped; deleted rows additionally show strikethrough with a red marker.
  • Bulk archive by scout (aged-out workflow): the Lookups & Admin screen now has a per-scout Mark aged-out button. Confirms the action, then archives all of that scout’s ledger entries in one pass with archivedReason: scout_aged_out and flips scout.active = false. A Reactivate & restore button reverses it, restoring only the entries archived for that reason — entries archived manually stay archived.
  • Dashboard counts reflect active entries only: the sidebar badge and the “Ledger Entries” stat now exclude archived and deleted rows. Recent Activity on the dashboard skips them too.

Prototype Seed Data

  • Client-side scale generator: advancement-leader.js now synthesizes ~5,000 deterministic ledger entries on load (seeded LCG, repeatable across reloads) so pagination, sorting, and archive can be exercised at realistic volume without bloating data/advancement.json. Clearly marked PROTOTYPE SEED in the code — remove the single generateSeedLedger(5000) call (and the function itself) when wiring real data.
v0.6 May 24, 2026 New

Advancement Section — Public Pages

  • New page: advancement.html — troop-wide Advancement landing. Stats strip (active scouts, MBs earned, camping nights, service hours, Eagle MBs), rank-distribution grid, and a sortable/filterable roster of every active scout with rank, MB count, nights, and last activity. Sort by any column, filter by name/patrol/rank. Click a row to drill into the scout’s Clipboard.
  • New page: advancement-scout.html?id=<scout_id> — per-scout dashboard recreating the existing Clipboard PDF as a live page. Rank-progression timeline, 6-metric summary, three-column dense rank grid (Scout → Life), Merit Badges (Eagle vs Other), Activities, Leadership, and Service Projects. Maya Sankpal-Tatera’s record is fully encoded from her actual Clipboard PDF (~155 rank-requirement rows across 7 rank sections, plus 25 merit badges, 19 activities, 2 SPL terms, 4 service projects). Other scouts are illustrative stubs. Print PDF button strips chrome for hand-out copies at Boards of Review.
  • Existing nav wired up: The previously-stub Advancement nav item on Home, Meeting, Calendar, and the Home sidebar Advancement Tracker teaser now all link to advancement.html.

Leader Workspace (Admin)

  • New admin app: admin/advancement-leader.html — behind the existing login overlay. A single-page workspace with sub-nav for Dashboard, Fast Entry, Event Roster, Universal Ledger, Court of Honor, Scoutbook Export, and Lookups & Admin.
  • Fast Entry — two equivalent workflows: scout-first (one scout, multiple requirements) and requirement-first bulk entry (one requirement, many scouts, checkbox grid). Both write into the in-memory ledger and surface a toast confirming the count.
  • Event Roster Entry: pick an activity, apply per-row attendance, camping nights, hiking miles, service hours, and free-text requirement codes. “Apply Defaults to All” cascades event-level defaults to attending scouts.
  • Universal Ledger: filterable table of every entry. Soft-delete with confirmation (the prototype shows the audit-preservation behavior).
  • Court of Honor Prep: COH metadata + candidate list since the previous COH. Toggle per-row include, see summary chips (recognitions / ranks / merit badges / leadership terms) update live, and view recent COH history.
  • Scoutbook Export: pipe-delimited CSV preview (MemberID|FirstName|MiddleName|LastName|AdvancementType|AdvancementID|Version|DateCompleted|DateApproved|DateAwarded), with Copy and Download. Rows missing a BSA Member ID are flagged and excluded.
  • Lookups & Admin: editable views of internal requirement codes, scouts & BSA IDs, adult-leader signoff initials, and the merit badge catalog.

Data & Architecture

  • New data file: data/advancement.json — ledger-driven schema following the GiaChiPT spec. Contains rank taxonomy with sub-requirements (first aid subtasks, fitness sub-steps, lashings), merit badge catalog with Eagle flags and Scoutbook IDs, leader signoff registry, 9 scouts (Maya at full fidelity), and pre-built COH candidates + Scoutbook export preview.
  • Internal taxonomy preserved: requirement records keep Troop 79’s compact internal codes alongside the official BSA mappings — the official hierarchy never overrides the operational tracking structure.
  • Prototype constraints: All mutations (fast entry, roster save, COH inclusion, soft delete) are in-memory only. No persistence layer yet — Supabase + Next.js per the spec is the production target.
v0.5 April 9, 2026 Feature

Meeting Archive & Multi-Meeting Editor

  • Multi-meeting archive: The editor now manages an ongoing record of every meeting in a single meetings.json array, replacing the single-meeting meeting.json format. All meetings persist in localStorage between sessions and are exported together as one file.
  • Meetings dashboard: The editor opens to a table of all meetings (sorted newest first) with date, type badge, agenda summary, and per-row actions: Edit, Copy, Preview, Delete.
  • Copy meeting: Duplicates any past meeting and auto-bumps the date forward 7 days — designed for the weekly workflow of creating next Sunday’s meeting from last week’s template.
  • Preview any meeting: Preview button on each row opens the public meeting page with that specific meeting’s data, regardless of date.
  • Public meeting page updated: meeting.html now loads from meetings.json (array), picks the next upcoming meeting automatically, and supports ?date=YYYY-MM-DD to view any specific past meeting. Backwards-compatible with legacy single-meeting format.
  • Sample data: data/meetings.json seeded with 3 meetings from the March 8, 15, and 22 Bugle issues.
v0.4 April 9, 2026 Feature

This Week’s Meeting Page

  • New page: meeting.html — public-facing meeting display answering “What’s happening this Sunday?” for parents and scouts.
  • At-a-Glance card: Large date, time, color-coded uniform badge (navy = Class A, forest = Class B), Google Maps link for location, snack and patrol assignments, duty roster link.
  • Agenda timeline: Vertical timeline with time column and parallel track detection — when younger and older scout activities share the same time slot, they render side-by-side. Stacks on mobile. Track badges: Open Advancement (forest), Merit Badge (bark), Event Prep (navy), Special (khaki). Scout names shown as first name + last initial.
  • Pre-meeting section: Conditional rendering with bark accent border for early advancement sessions and PLC meetings.
  • No-meeting state: Full page replacement with centered notice, reason, and next meeting date.
  • Stale data detection: Automatic banner when the meeting date has passed.

Meeting Data Editor

  • New admin tool: admin/meeting-editor.html — form-based editor for creating and editing meeting JSON data. Accessible from “Members Login” on all pages.
  • 6 collapsible form sections: Meeting Details (with no-meeting toggle), Pre-Meeting Activities, Agenda (dynamic items with scouts/requirements sub-lists and reorder), Announcements, Looking Ahead, Bugle Settings.
  • Actions: Save Draft, Load File, Preview (opens meeting page in new tab), Copy JSON, Download JSON. Autosave every 30 seconds.
  • Shared admin infrastructure: editor-shared.css and editor-shared.js provide reusable styles and utilities (file I/O, dynamic lists, localStorage drafts, toasts, collapsibles, login gate) for future editors.

  • Approved Duty roster: Public. Linked on the meeting page.
  • Approved Patrol names: Fire Quacker, Shooting Star, Screaming Eagles, Buff Burritos. Defined as config arrays in the editor JS.
  • Approved Track categories: Open Advancement, Merit Badge, Event Prep, Special.
  • Approved Deployment: Vercel (static hosting with Git-based deploy). Editor exports JSON for commit/push workflow.
  • Approved Long-term vision: Reverse the workflow — publish to the website first, then auto-generate the Bugle email from the website data. Data model designed to be rich enough for future Bugle generation.
  • Approved Multi-meeting archive: All meetings stored in a single meetings.json array. Copy-from-previous workflow for weekly meeting creation.
v0.3 April 9, 2026 Feature

Events & Calendar Page

  • New page: calendar.html — full calendar view with events grouped by month, matching the site’s NYT-style editorial design. Events are color-coded by type (9 categories), with dimmed “No Meeting” rows and highlighted multi-day events (campouts, high adventure, summer camp).

  • Homepage sidebar events now dynamically populated from calendar.json, showing the next 4 upcoming non-cancelled events. Previously used hardcoded placeholder events. Events nav link now points to calendar.html.

Content Source Analysis

  • Analyzed 3 recent Troop 79 Bugle emails (March 7, 15, 21) to identify consistent content patterns for the calendar, meeting agenda, and advancement sections. Patterns documented and used to define data models for the calendar (this release) and meeting page (upcoming).
