What Was Accomplished

Has/Needs Tool (new)

Built a new admin screen at /admin/advancement/has-needs, linked under Planning: a leader checks one or more Scout / Tenderfoot / Second Class / First Class requirements and instantly sees who already Has them and who still Needs them, sorted by current rank then first name — a meeting-prep aid for lining up a station or a signing-off session.

Shipped in two passes. V1 (v1.6.0) launched with an AND/OR combine toggle for multiple checked requirements. Follow-up feedback simplified it: a Clear-all button was added, the toggle was removed in favor of one fixed rule (Has = completed every checked item, Needs = completed none), and scouts with partial progress now get their own italicized "(Partially Complete)" line at the bottom of the Needs column instead of being silently dropped or folded into either bucket.

Design was checked against ux-lead before building — the recommendation to keep checkboxes leaf-only (not on parent/group rows) avoided layering a second all/any/n-of rule on top of the combine logic, consistent with this project's standing "simplify, don't layer" pattern. qa-lead reviewed the implementation post-build: PASS, with one minor pagination-consistency nit (fixed) on the rank-requirements query.

Fast Entry: two UX fixes

Scout-First Entry's "N pending" notice used to pop in above the rank tabs the moment a requirement was checked, shoving the whole card down a row. It now renders inline next to the Clear/Save buttons — no more layout jump. Separately, the Save button's text was rendering visibly smaller than Clear's; the root cause was that .btnPrimary is used standalone everywhere in this codebase (never combined with .btn), so it was silently missing font-size, font-weight, and padding, falling back to the browser's default button styling. Fixed by merging the shared typography onto a combined selector — corrects every button using .btnPrimary, not just the one flagged.

Decisions

[D-015] Has/Needs Tool: leaf-only checkboxes, all-or-none combine logic, partial shown separately. Only leaf requirements are checkable (checking a parent group would layer a second completion rule on top of the combine logic). The AND/OR toggle was removed in favor of a single fixed rule: Has = completed every checked item, Needs = completed none of them. Partially-complete scouts are listed separately — italicized, "(Partially Complete)" — rather than hidden or miscategorized. Reuses lib/mb-helpers.ts's buildReqTree, matching the meeting-plan engine and mb-progress precedent rather than reaching into fast-entry's picker-scoped helpers.
