What Was Accomplished

The reported bug — Robotics requirements missing in Fast Entry

A leader couldn’t sign a scout off on the Robotics merit badge — Fast Entry showed zero requirements. The badge data was fine (Robotics has all 23 requirements in production). The loader fetched the entire merit_badge_requirements table (now 1,735 rows) in one request, and PostgREST silently caps a response at 1,000 rows — everything alphabetically past leatherwork was dropped, hiding 31 badges’ requirement trees. Fixed fast-entry/page.tsx and the lookups MB-tree editor with the existing fetchAllRows() paginator. Verified live against production before and after: unbounded fetch returned 1,000 rows / 0 Robotics; paginated fetch returns all 23. Shipped as v1.9.1.

Follow-up audit — the whole read surface

Scanned every read of the only two tables that can exceed 1,000 rows — ledger_entries/ledger_active (9,388) and merit_badge_requirements (1,735); every other table is ≤171 rows. Checked each unpaginated read against live counts and preventively paginated three more before any could truncate:
