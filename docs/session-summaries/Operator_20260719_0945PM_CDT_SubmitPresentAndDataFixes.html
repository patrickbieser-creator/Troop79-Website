What Was Accomplished

Submit & Present

New admin screen tracks two independent human confirmations per rank / merit badge / special award — submitted to Scoutbook, and presented to the scout — each recording who and when, following the same audit-column pattern already used for archive/delete. Scoutbook Export gained a one-click “mark previewed rows as submitted” bulk action for right after an upload is verified.

Login Access-Control Fix

Found live in production: the shared scout-role login could read 13 leader-only admin pages — the full ledger, meeting attendance, duplicate-record audits, fast entry, lookups, and the dashboard. Only Server Actions were role-gated; the pages themselves only checked “is authenticated.”

All 13 pages fixed with the same requireRole(['leader']) pattern already used elsewhere. Added a second, independent layer — an edge-level allowlist in proxy.ts — so a future new page can’t reopen the same class of gap just because someone forgot the per-page check. Scout access was explicitly widened to the Has/Needs Tool by request.

Merit Badge Data-Correctness Fixes

Found and fixed the same underlying bug in three independent places, all caused by extracting a merit badge id from a ledger code by splitting on the first hyphen — which breaks for the 30 of 64 badge ids (First Aid, all four Citizenship badges, Emergency Preparedness, etc.) that contain a hyphen themselves:

  • The mb_progress SQL view — silently dropped “in progress” counts on both the public and admin Merit Badge Progress catalogs, and fed wrong candidate data to the Meeting Plan engine, a third non-obvious consumer of the same view.
  • Fast Entry’s keyForLedgerRow — silently disabled Scout-First’s own duplicate-protection for those same 30 badges.
  • /merit-badges and /advancement — both public pages used no Dynamic API, so Next.js silently prerendered them as static HTML at build time. Neither could reflect any database change, including the mb_progress fix above, until the next deploy rebuilt them.
Verified end-to-end against real data on both local dev and production before shipping: Archery went from showing “0 in progress” to correctly showing “1 in progress” live on www.troop-79.com.

Fast Entry Duplicate Prevention

Requirement-First’s bulk multi-scout entry had no protection against re-signing a requirement a scout already had — a structural gap (scout selection happens after requirement selection, so there’s no per-scout completion context at pick time) with no database constraint behind it either. Very likely a real contributor to the ~434 duplicate ledger record groups already sitting in the Audits backlog.

Added a server-side hard block (skips existing scout+kind+code matches among the four one-time/binary kinds, reports what was skipped) plus a pre-submit warning in the confirm dialog. Verified live: bulk-signing a requirement for two scouts, one of whom already had it, correctly saved 1 and skipped 1 with no duplicate row created.

Lint Baseline Cleanup

10 long-standing errors (unrelated to this session’s feature work) fixed to 0 — prefer-const, explicit any in the seed script, an <a> that should have been <Link>, a dead variable, and three scoped suppressions for legitimate setState-in-effect patterns. Two remaining warnings left deliberately — both need a real refactor with visual verification, not a mechanical sweep.

Decisions

  • [D-037] Scout/leader access must be gated at the page AND the edge, not just the Server Action.
  • [D-038] Submit & Present reuses the existing audit-column pattern on ledger_entries, not a new table.
  • [D-039] Resolve merit-badge ids by matching real catalog rows, never by splitting on hyphen position.
  • [D-040] Public Server Component pages must explicitly force dynamic rendering when they show live data.
  • [D-041] Fast Entry bulk actions get a server-side dedup backstop, not just UI-level protection.
