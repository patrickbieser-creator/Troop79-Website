Event Signup went from a plan document to live in production in one session. Families can now sign up for calendar events online: who’s coming, what they owe, who can drive, which jobs they’ll take, and what they’re bringing to a potluck.

What Was Accomplished

From plan to schema

Started by grounding against production and closing the two decisions that gated the migration (guest model, calendar category mapping). Wrote the Phase 1 schema — ten tables, six RPCs — and verified it with 13 constraint and concurrency assertions rather than assuming it worked: the identity CHECK, driver-only and contributor rules, per-leg seat constraints, and the partial unique indexes that make the D-023 duplicate-row failure impossible by construction (a live duplicate is rejected; a re-signup after cancelling is allowed).

The design pivot that shaped the feature

Patrick redirected the shift signup partway through: instead of listing every job under every family member, lead with the work. That became D-029 — one job board, grouped by day, where the gate meets a visitor at the row they clicked rather than at the bottom of a 30-job page. The underlying storage never changed; only the input surface inverted, so roster and owed math stayed person-indexed throughout.

A later report exposed the assumption underneath it: an Eagle Court of Honor with three potluck tasks showed families nothing, because the job list only rendered when attendance was off. Events genuinely need both at once (D-034), and the wording had to bend too — “shifts & tasks” reads correctly for a pancake breakfast and badly for “bring a dessert”.

Leader tooling

  • Event Signups — a block checklist seeded by category, with editable jobs, tiers and questions, and a guarded remove-signup path whose first press is a dry run reporting how many people would lose their signup.
  • Event Rosters — troop-wide totals, two-deep indicator (driver-only correctly excluded), per-leg driver seats, job coverage, owed vs. paid, non-responder chase list, print and CSV, and remove-with-restore for the families who phone rather than use the form.
  • Calendar — Upcoming/Past tabs, oldest-first, and Clone for turning a configured troop meeting into next month’s.

Blockers Resolved

ProblemDiagnosis
Family gate appeared to do nothing on sign-outIt worked — the gate checks the leader session first, so deleting the family cookie changed nothing visible for a signed-in leader. The button now only appears for actual family sessions, and confirms when it fires.
Nested <form> hydration errorThe sign-out bar was its own form and ended up inside the submit form. The standalone-board design existed precisely to avoid this and the status bar walked into it from the other side.
Court of Honor tasks invisible to familiesJobs only rendered when attendance was off; Ceremony defaults it on. Three tasks existed in the database with no path to claim them.
Public job list lost its paddingAn unscoped .slotList > li { padding: 0 } added for the form overrode the read-only list, which has no inner button to supply its own.

Review Findings

Two qa-lead passes each caught something real in code that had already been convinced-of:

  • Open redirect — next.startsWith('/') does not stop /\evil.com, which the URL parser resolves off-site. On a password form that is a phishing redirect aimed at the troop password. Replaced with origin comparison (12 assertions), and the identical pre-existing flaw in the admin login was fixed by the same helper.
  • Email PII leak — the whole recipient list went in a shared to:, so the first “Send” would have shown every parent every other parent’s address. Unrecallable. Now one message per recipient.
  • Scout-role exposure — the new admin pages relied on a session check a shared-password scout login satisfies, which would have exposed guest notes, driving arrangements and payment status. All four are now leader-only.

Deployment

Code was deployed before the migration, deliberately: main’s old code had no category-colour fallback, so migrating first would have returned a 500 on /events for every visitor until Vercel finished. Code-first meant the worst case was briefly-grey category chips.
