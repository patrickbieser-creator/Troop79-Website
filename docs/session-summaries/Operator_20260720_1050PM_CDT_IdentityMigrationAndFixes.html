What Was Accomplished

Signup identity closes the loop (D-048)

The public signup picker stopped showing anyone twice back in v1.14.0, but the signup_entries record underneath still recognized a participant by whichever legacy column they signed up through — scout_id, scout_parent_id, or leader_code — not by who they actually are. A person reachable both ways (a parent who is also a troop leader) could still land two signup rows for the same event.

Seven migrations, deliberately sequenced so each one is safe to ship on its own: fix the one write path that still created a person without an identity record; have the RPC write person_id before it ever reads it (split across two separate migrations, so a row still mid-flight is never invisible to the lookup); a guarded backfill and NOT NULL constraint that verify their own precondition rather than trust an earlier step; the existing-row lookup flips to prefer the real identity last.

Found along the way: the original identity-spine backfill (shipped weeks earlier) had silently merged three parents into their own children's identity record, whenever a minor's contact email happened to be the parent's own (Porter, Kimble, Barry families). Confirmed present on production via a full dry run against a real data snapshot, pulled down specifically to test this before ever touching prod — then fixed and verified there were zero remaining instances of the pattern anywhere in the system.

First automated tests in this project (D-049)

Vitest, integration-style against a real local Postgres instance — no database mocking. Chosen over pgTAP on tech-lead's recommendation: every target behavior is reachable through the exact call path production code already uses, and a second SQL-native test runner wasn't worth standing up for a solo-dev project with no CI. Six tests now cover the dual-identity dedup fix, two D-033 capacity/waitlist regressions, and the merge_people fix below.

merge_people closes the same gap through a second door (D-050)

A qa-lead review of the identity migration caught something the migration itself couldn't see: the existing admin “merge two duplicate people” tool never moved signup_entries.person_id — a routine merge could have silently reopened the exact bug the whole migration exists to close. Fixed to reassign correctly, and to block the merge outright (nothing written) if both people already hold a live signup for the same event — that's a genuine conflict for a human to look at, not something safe to guess.

Verified for real: Patrick performed an actual merge (a genuine duplicate “Summer Kimble” record) through the live admin tool after deploy, and every downstream link — both scout_parents rows, both relationship edges — reassigned correctly to the survivor.

Three unrelated production bugs, found and fixed live

None of these were part of the plan going in — each surfaced from real use of the live site during or right after the identity work, and each was fixed and verified on production the same session.

Decisions

  • D-048 — submit_household_signup migrated to person_id via a write-before-read migration sequence, closing D-042/D-047.
  • D-049 — Vitest, integration-style, chosen as this project's first test framework over pgTAP.
  • D-050 — merge_people now reassigns signup identity and blocks on live-signup conflicts.
  • D-051 — RLS enabled with zero policies on the 9 people-spine tables (B-001 fix).
  • D-052 — Scout rank auto-awards on completing all requirements, mirroring the existing BoR trigger.
  • D-053 — photo_albums.category re-synced to the current calendar taxonomy.
Deployment discipline this session: every migration was dry-run tested against a real production data snapshot before ever touching production, and every fix was verified live on the actual site (browser-tested, not just query-checked) immediately after deploy. Nothing shipped on faith.
