What Was Accomplished

Families can now update their own scout's info

A new public /profile page — reached from the utility bar, deliberately its own surface rather than a step bolted onto Event Signup's household picker — lets a family sign in with the shared troop password and submit a proposed update to their scout's address, phone, email, school, grade, swim classification, birthdate, or the new "Things We Should Know" field.

Nothing a family submits touches the live record. Proposals land in a new change_requests table and only apply once a leader approves them from a new diff panel folded into the existing Scout editor. A second submission overwrites the first pending one — with an in-UI warning shown first — rather than queuing.

The browser stays "logged in" as a household across visits via a second signed cookie layered on top of the existing family session — additive, doesn't change what that cookie proves, doesn't touch Event Signup's separate flow. A Log out control fully resets the session so a shared device can hand off to a different family.

New field: "Things We Should Know"

Scout and adult records now have a freeform place for food allergies, medical conditions, and special needs — visible to leaders only, and one of the fields a family can propose an update to. This reverses a prior explicit decision (D-014) to store no medical data, by the project owner's direct sign-off mid-session, specifically because the review-approval gate above is the real safeguard, not gatekeeping which fields families can touch.

Leaders: a new "Needs Attention" panel on the Dashboard

The Leader Dashboard now opens with a list of anything waiting on a decision, instead of leaders needing to already know which scout to check. Clicking an item deep-links straight to that scout's editor with the record already open (?tab=active_scout&open=<id>).

Built as an extensible list of categories, not a single hardcoded item type — submitted news articles and submitted site suggestions will land in the same space later without changing the panel itself.

Fixes & polish

  • The Scout editor's Parents/Guardians section was pulling styles from the wrong CSS module and rendering completely unstyled — broken grid, plain-text dropdown, an oversized Unlink button. Root-caused and fixed.
  • Button-sizing and layout polish from a ux-lead review: a distinct Promote-to-adult style, a collapsible "+ Add parent/guardian" card instead of an always-open control strip, proper label associations.
  • Active/Inactive status is now the same two-option radio toggle everywhere (scouts, leaders, adults) — previously two different UI patterns.
  • The household picker dropdown no longer trails every member's name after the household label.

Deployment

Standard mid-session pause: production deploys are a mandatory escalation trigger in this project's rules, so an Opus risk assessment ran before touching anything live. Sequence followed its recommendation:

  • Production DB backed up (schema + data dump) before migrating
  • Migrations applied to production first (additive-only — new nullable columns, a new table — compatible with already-live code)
  • RLS verified live via a direct anon-key REST probe against both new surfaces — not assumed from the migration
  • Code committed, pushed, Vercel auto-deploy confirmed Ready
  • Smoke-tested live: homepage 200, /profile 200, Profile link visually confirmed in the real utility bar
