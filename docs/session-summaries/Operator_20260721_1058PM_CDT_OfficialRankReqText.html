What Was Accomplished

New reference field, built to last

Added a requirement_official_text table so leaders can keep the verbatim official BSA wording for any rank requirement on file, right alongside the troop's own internal code/label catalog — without touching either existing catalog table. Keyed by (source, parent_id, code) rather than the requirement row's own id, specifically because the Merit Badge editor fully rebuilds its requirement rows (fresh ids) on every save — an id-based link would have silently lost data the first time a badge was edited.

Security posture: the new table ships with Row-Level Security enabled and zero policies — the same "service-role only" pattern already used for sensitive tables in this project. The anon key cannot read it under any circumstance.

Internal Requirement Codes editor, extended

The admin editor now surfaces every rank requirement row, including nested sub-requirements (like Tenderfoot's lettered 4a.1–4a.8) that had no edit screen at all before this session. Each row gets a new "Official text" field, saved through the existing rename-safe code/label action.

Content sourced from the leader, not the internet

Rather than auto-fetching BSA's copyrighted requirement text, the actual content came from Patrick pasting and uploading the real source material (typed text, a PDF screenshot, and a full rank-requirements-book excerpt) over the course of the session — mirroring the project's existing paste-driven pattern for this exact category of content. Merit badges (69 badges, ~1,900 rows) were explicitly left out of this pass.

A real catalog gap, found along the way

While reviewing the Tenderfoot text against a current edition, Patrick caught that his first upload was an older printing — three wording differences, plus one entirely new requirement (5d, on hiking durable surfaces) that isn't in the catalog at all yet. Added as a new row, positioned correctly, text imported.

Worth checking: only Tenderfoot happened to get compared against a current edition. The other 6 ranks were seeded the same way, at the same time — they may have the same kind of gap, just undiscovered.

Shipped to production, correctly

Since all of the actual text lived only in local dev (inserted via a one-off script, not a migration), deploying meant more than a code push: the schema migration went to production first, then every rank's text — and the new Tenderfoot row — was explicitly replicated from local to prod using the same import script pointed at production credentials, so the two environments would actually match rather than just sharing an empty table.

Decisions

  • D-056: Official text keyed by natural key (source/parent/code), not a bigserial foreign key — avoids silent data loss on catalog re-saves.
  • D-057: Content population stays paste-driven, never AI-scraped, for both ranks and merit badges — merit badges intentionally left unpopulated this session.
  • D-058: The requirement-codes editor now shows every rank row (not just top-level) rather than building a full nested-tree editor just for this feature.
