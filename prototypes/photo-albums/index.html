Prototype Annotations & Design-to-Dev Handoff

1. Navigation Placement — Recommendation

  • Recommendation: "Photos" as its own top-level tab, placed between Calendar and Advancement (rendered above).
  • Why top-level: photos are the highest-emotion, highest-repeat-visit content for parents — the single most likely reason a family opens the site on a phone after a campout. Burying 60 albums under About (a join-oriented page) or footer-only would make the site's richest content its least discoverable.
  • Why next to Calendar: mental model is "events happen on the Calendar, photos of them land in Photos." The pairing reinforces the event-to-album relationship and matches the existing calendar category taxonomy reused here.
  • Nav width check: six tabs at 13px/16px-padding fits well under the 1180px shell alongside the search box; verified no wrap at 768px (hamburger takes over below 600px, consistent with index.html).
  • Secondary placements to add in production: a "Photo Albums" link in the footer Navigate column, and a "Latest album" module candidate for the homepage sidebar (park as a follow-up idea).

2. Key UX Decisions

  • Year grouping as the spine, filters as the lens. With 50–60 albums, reverse-chronological year groups (~12/year) match how families remember events ("that was summer camp two years ago"). Category chips + year chips + search combine with AND logic; year headings hide when a group empties.
  • Featured "Latest Albums" strip (3 newest) serves the dominant use case — "show me last weekend's photos" — without scrolling. It hides whenever any filter/search is active so the same album never appears twice on a filtered view.
  • Whole card is the link (single <a target="_blank">) — biggest possible touch target for phones. No nested links inside cards.
  • External-destination affordance is persistent, not hover-only: a "Google Photos" badge sits on every cover (reusing the news-cms gl-badge Gallery Link pattern and its arrow glyph), plus an arrow-icon "Google Photos" item in the meta line, plus the page dek states it in prose. Hover-only affordances fail on touch devices.
  • Sticky filter bar (below the sticky nav) so filters stay reachable while scrolling a long grid. Chip rows scroll horizontally on mobile instead of wrapping — keeps the sticky header short so it never eats half a phone screen.
  • Mobile is 2-up compact: descriptions drop, type tightens. Density beats detail when a parent is thumb-scrolling 60 covers; the title + date + cover is the recognition unit.
  • Category chips show counts ("Campout (7)") — sets expectations before tapping and exposes empty categories honestly instead of hiding them.

3. States Considered

  • Empty filter result: friendly empty state + one-tap "Clear All Filters" (try Category: High Adventure + Year: 2026).
  • Empty category: chips for categories with zero albums are still rendered with "(0)" and produce the empty state rather than disappearing — predictable UI beats a shape-shifting filter row. (All 8 categories have data in this sample; delete the Meeting albums from the data array to see it.)
  • Missing cover photo: khaki monogram fallback ("79 / Troop Album") — see "Winter Cabin Campout 2025" and "Popcorn Kickoff 2024". Production rule: never render a broken-image icon; fall back on onerror too, since Google-hosted cover URLs can expire.
  • Missing description: card compresses gracefully; meta line pins to the card bottom via flex so meta rows still align across a grid row.
  • Missing photo count: meta line simply omits it (see Winter Cabin Campout) — no "0 photos" lies.
  • Very long title: clamped at 3 lines with ellipsis; the full title is preserved in the card's aria-label and title attribute (see the Summer Camp Week Two 2024 album).
  • Loading state (production): reuse the site's existing .skeleton shimmer as card-shaped placeholders — not built here since the prototype renders synchronously.
  • Zero albums at all (production): same empty-state component, copy: "Albums are coming soon — check back after our next outing."

4. Accessibility Notes (WCAG 2.1 AA)

  • Filter chips are real <button>s with aria-pressed; chip groups have role="group" + labels. Selected state = navy fill (7.6:1 on white text), not color alone — pressed state is also programmatic.
  • Each card link's aria-label reads: title, category, date, photo count, "Opens Google Photos in a new tab" — screen-reader users get the external-destination warning that sighted users get from the badge.
  • Results count is an aria-live="polite" region — filter changes are announced.
  • Full keyboard support: chips and cards in DOM order, visible :focus-visible outlines (forest, 2px, offset), no positive tabindex, Escape not needed (no modals).
  • prefers-reduced-motion kills the card lift/transitions.
  • Heading order: h1 page title → h2 year headings → h3 album titles.
  • Contrast: body/meta text on newsprint pass AA; the khaki Court of Honor tag uses navy text (5.1:1). Known site-wide item: white-on-bark tags (~3.9:1) match the existing site's tag treatment — flagged for a site-wide token review rather than diverging on this page.

5. Production Build Needs

  • Data model — photo_albums table (Supabase): id, title, event_date (date; display month/year), category (reuse the calendar kind enum — do NOT create a parallel category lookup, per the "kind alone carries event category" decision), google_photos_url, cover_image_url (nullable), description (nullable), photo_count (nullable int), is_featured (optional override; default = 3 newest), published.
  • Cover images: Google Photos image URLs are not stable long-term. Recommend uploading one chosen cover per album to Bunny CDN (already in the stack for News) and storing that URL — one image per album is cheap and reliable.
  • Admin needs: simple CRUD in the existing admin workspace — paste share URL, title, date, category, optional description/count, upload cover. Nice-to-have: paste-URL-first flow since the share link is what the volunteer has in hand.
  • Photo counts go stale (shared albums grow). Either label it "200+ photos", or drop the count field entirely, or accept manual refresh. Decision needed — see open questions.
  • Filter state in URL (?category=campout&year=2025) so filtered views are shareable in the Bugle newsletter.
  • Server-render the full list (60 items is small); client JS only for filtering, same logic as this prototype.

6. Open Questions for the Team

  • Photo counts: display at all, given they drift as shared albums grow?
  • Are all 50–60 album share links truly public ("anyone with link")? Any albums with faces/names that should be members-only? If some are, does this page need a login-gated tier?
  • Should the homepage sidebar get a "Latest Album" module linking here? (Parked as a plan-level idea, not built.)
  • Multi-year events (e.g. a New Year's lock-in): group by event_date year — confirm that's acceptable.
  • Who owns backfilling the ~55 historical albums into admin — worth a one-shot import script from a spreadsheet, like the ledger backfill?
